# Agent Foundation Template Authoring Guide

## Purpose

This guide explains how template roots, generated outputs, and ownership rules
work in the npm package.

Use it when:

- adding a new base or overlay output
- adding a new constraint or practice profile
- deciding whether a file should be managed or merge-managed
- updating marker structure

## Important Implementation Detail

The package no longer relies on template file bodies alone for final content.

Template roots now act as:

- output inventory
- ownership declarations
- profile-to-output mapping anchors

Most repository document content is generated dynamically by
`npm/src/lib/foundation-content.ts` using:

- repository scan facts
- manifest selections
- project context answers

That means authoring a new template root is necessary, but it is not sufficient
on its own. You usually also need to update generated-content logic.

## Template Roots

Templates live under `npm/templates`.

### Base roots

Always installed:

- `templates/base/AGENTS.md`
- `templates/base/CLAUDE.md`
- `templates/base/GEMINI.md`
- `templates/base/.agent-foundation/handoffs/current.md`
- `templates/base/.agent-foundation/handoff/design-ready.md`
- `templates/base/.agents/skills/...`
- `templates/base/docs/agents/repo-facts.md`
- `templates/base/docs/agents/design-handoff.md`
- `templates/base/docs/product/index.md`
- `templates/base/docs/architecture/overview.md`
- `templates/base/docs/engineering/index.md`
- `templates/base/docs/rules/index.md`
- `templates/base/docs/skills/index.md`

### Overlay roots

Installed only when selected:

- `templates/frontend/<profile>/...`
- `templates/backend/<profile>/...`
- `templates/system/<profile>/...`
- `templates/quality/<profile>/...`
- `templates/constraints/<profile>/...`
- `templates/practice/<profile>/...`

Examples:

- `templates/frontend/next/docs/architecture/frontend.md`
- `templates/backend/nest/docs/architecture/backend.md`
- `templates/system/internal-tool/docs/system/overview.md`
- `templates/quality/ci-basic/.github/workflows/ci.yml`
- `templates/constraints/payments/docs/product/constraints/payments.md`
- `templates/practice/ddd-core/docs/practices/ddd-core.md`

## Path Mapping Rules

Repository paths are derived from template paths by removing the layer prefix.

Examples:

- `templates/base/docs/rules/coding.md` -> `docs/rules/coding.md`
- `templates/frontend/next/docs/architecture/frontend.md` ->
  `docs/architecture/frontend.md`
- `templates/constraints/payments/docs/product/constraints/payments.md` ->
  `docs/product/constraints/payments.md`
- `templates/practice/tdd-first/docs/practices/tdd-first.md` ->
  `docs/practices/tdd-first.md`

## Ownership Rules

Every output must have a deliberate ownership class.

### Managed

Use `managed` when the package should fully rewrite the file on `sync`.

Current examples:

- `.agent-foundation/manifest.json`
- `.agent-foundation/profile-registry.json`
- `.agent-foundation/context-budget.json`
- `.agent-foundation/provider-projections.json`
- `.agent-foundation/doc-index.json`
- `.agent-foundation/doc-health.json`
- `.agent-foundation/handoffs/current.md`
- `.agent-foundation/handoff/design-ready.md`
- `.agents/skills/*`
- `docs/agents/repo-facts.md`
- `.github/workflows/ci.yml`

### Merge-managed

Use `merge-managed` when the package should own only stable sections and leave
the surrounding document editable.

Current examples:

- `AGENTS.md`
- `CLAUDE.md`
- `GEMINI.md`
- `docs/product/problem-and-users.md`
- `docs/architecture/frontend.md`
- `docs/engineering/testing-strategy.md`
- `docs/product/constraints/*.md`
- `docs/practices/*.md`

### User-owned

Do not create a template for files the package should never rewrite.

## Marker Rules

All merge-managed files must use:

```md
<!-- agent-foundation:begin section="section-name" -->
Managed content here.
<!-- agent-foundation:end section="section-name" -->
```

Rules:

- begin and end markers must match
- nested markers are invalid
- content outside markers is user-owned
- keep section ordering stable where possible

## Generated Content Rules

When adding or changing docs generated by `foundation-content.ts`:

- keep root entry files concise
- keep provider projection files semantically aligned with the same canonical docs
- prefer stable headings and phrasing so diffs stay reviewable
- separate facts from assumptions
- reflect selected constraints and practice profiles in the resulting docs
- keep handoff instructions procedural and short

## New Profile Checklist

When adding a profile:

1. Add the template root under `npm/templates`.
2. Register it in `npm/src/lib/profile-registry.ts`.
3. Update output ownership in `npm/src/lib/file-ownership.ts`.
4. Update `foundation-content.ts` if the profile changes generated content.
5. Update prompt options if the profile should be selectable in interactive init.
6. Add or update tests for:
   - template resolution
   - init
   - sync
   - doctor

## Testing Expectations

Every new template or generated-output change should be covered by at least one
of:

- prompt/selection tests
- template resolution tests
- init generation tests
- sync behavior tests
- doctor validation tests

If a new file is merge-managed, add both:

- a sync test that proves the managed block is restored
- a doctor test that proves corrupted markers are detected
